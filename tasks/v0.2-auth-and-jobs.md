# Task: v0.2 — Auth (Google) & Job Creation

**Phase:** v0.2 — Авторизация и создание задач
**Status:** not started
**Depends on:** v0.1 — SEO core (done)

## Goal

Add Google OAuth login via Supabase Auth. Authenticated users can submit YouTube URLs for transcription from a personal dashboard. Public pages remain accessible without auth.

## Design Decisions

- **Auth provider:** Google OAuth only (no email/password).
- **Post-login redirect:** `/dashboard`.
- **Login page:** dedicated `/login` page with a single "Sign in with Google" button.
- **Sign-in entry points:** header button (always visible) + CTA block on transcript pages.
- **Job form:** YouTube URL field + multi-language selector (EN always included, user picks additional languages).
- **Dashboard:** card-based layout (reuses `TranscriptCard` style) with status badges and progress.
- **Database:** new `profiles` table + `user_id` column on `transcripts`.

## Database Changes (new migration)

### New table: `profiles`

```sql
create table public.profiles (
  id              uuid primary key references auth.users (id) on delete cascade,
  display_name    text,
  avatar_url      text,
  preferred_languages text[] not null default '{en}',
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);
```

- RLS: users can `select` and `update` only their own row.
- Trigger on `auth.users` insert → auto-create `profiles` row (pull `display_name` and `avatar_url` from `raw_user_meta_data`).

### Alter table: `transcripts`

```sql
alter table public.transcripts
  add column user_id uuid references auth.users (id) on delete set null;
```

- Nullable (existing seed data has no author).
- RLS update: `insert` requires `auth.uid() = user_id`; public `select` stays unchanged.

## Auth Flow (step by step)

```
1. User clicks "Sign in with Google" (on /login or CTA)
2. Client calls supabase.auth.signInWithOAuth({ provider: 'google', redirectTo: '/auth/callback' })
3. Browser redirects to Google consent screen
4. Google redirects back to /auth/callback with code
5. /auth/callback route exchanges code for session (server-side)
6. Redirect to /dashboard
```
